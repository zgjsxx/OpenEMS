# OpenEMS 当前能力与后续路线

本文档用于快速说明当前系统已经支持什么，以及下一阶段建议优先补什么。

## 1. 当前系统定位

当前 OpenEMS 已经不只是“采集 Demo”，而是一个具备以下基础闭环的单站运维系统：

- 实时采集
- 实时数据库 RtDb
- 手动遥控遥调
- 告警检测与处理
- 历史采样与查询
- Web 运维后台
- PostgreSQL 配置中心
- Docker 化部署
- 基础自动控制策略

## 2. 当前已经支持的能力

### 2.1 采集与通讯

当前已经支持：

- Modbus TCP 采集
- IEC104 采集
- 遥测 / 遥信 / 遥控 / 遥调 点表管理
- PostgreSQL 作为运行时配置主源
- CSV 作为导入 / 导出 / 初始化格式

### 2.2 实时与控制

当前已经支持：

- RtDb 共享内存实时库
- 实时点位快照
- Web 遥控 / 遥调下发
- 控制命令状态查看
- 人工命令审计记录

### 2.3 告警

当前已经支持：

- 告警规则配置化
- PostgreSQL `alarm_rules`
- 活动告警 / 历史告警
- 告警确认 / 消音
- 告警审计

### 2.4 历史

当前已经支持：

- 历史采样进程
- TimescaleDB 历史写入
- Web 历史查询
- 按点位、时间范围查询历史数据

### 2.5 配置与后台

当前已经支持：

- 登录认证
- 角色权限
  - `viewer`
  - `operator`
  - `admin`
- 配置概览
- 配置管理
- 用户管理
- 审计日志
- 系统监控
- 拓扑管理

### 2.6 拓扑管理

当前已经支持：

- `/topology` 拓扑页面
- SVG 单线图展示
- 节点 / 连线 / 点位绑定
- 实时值与告警联动显示

### 2.7 自动控制策略

这是当前新增且很关键的一块。

当前已经支持的自动控制能力包括：

1. 防逆流控制
2. SOC 上下限保护
3. 高 SOC 条件下的光伏限功率补偿

当前控制链是固定顺序：

```text
防逆流 -> SOC 保护 -> 光伏限功率补偿
```

当前的策略特点是：

- 以储能为主调节对象
- 当储能因高 SOC 无法继续吸收功率时
- 自动通过 `pv-target-power-limit` 限制光伏出力
- 支持人工接管优先

### 2.8 Docker 与 E2E

当前已经支持：

- Docker 部署
- PostgreSQL / TimescaleDB 容器化运行
- OpenEMS 主应用容器化运行
- BESS/PV/Grid 一体化模拟器容器
- 策略自动化 E2E 测试
- 模拟器页面 + CSV 场景导入的手工测试

## 3. 当前仍然存在的缺口

### 3.1 自动策略还不够完整

虽然已经具备基础策略闭环，但真正 EMS 常见能力还没有完全覆盖，例如：

- 低 SOC 禁止放电的完整 E2E 场景
- 手动接管 30 分钟的完整验证
- 储能停机时的光伏兜底限发
- 光伏限发恢复策略的长期稳定性验证
- 多储能 / 多光伏协同
- 需量控制
- 分时电价策略
- 削峰填谷
- PCS 功率限制

### 3.2 拓扑还缺编辑器

当前拓扑数据模型已经有了，但编辑仍偏“表结构配置”，还不是现场最习惯的拖拽式编辑体验。

### 3.3 配置版本能力还不强

当前还缺：

- 配置差异对比
- 配置版本回滚
- 配置发布记录
- 更明确的变更影响提示

### 3.4 控制安全链路还可以更强

当前还建议补：

- 更严格的二次确认
- 回读确认
- 控制失败原因诊断
- 命令互锁
- 更细粒度权限

## 4. 下一阶段建议优先级

### P0：把策略能力做成稳定可验收

建议优先做：

1. 低 SOC 禁止放电场景
2. PV 手动接管场景
3. BESS 停机 + PV 限发兜底场景
4. 光伏限发恢复场景
5. 策略页面配置体验增强

### P1：拓扑编辑器

建议做拖拽式拓扑编辑器，降低配置门槛。

### P2：配置版本管理

建议做：

- 配置 diff
- 配置回滚
- 配置版本号
- 配置发布记录

### P3：更完整的 EMS 策略

建议后续逐步扩展：

- 需量控制
- 削峰填谷
- 分时电价策略
- 光伏优先 / 储能优先切换
- 多设备协同仲裁

## 5. 当前最值得继续做的方向

如果只选一个方向继续往下走，我建议优先做：

**策略引擎能力补全 + 策略场景 E2E 扩展**

原因很直接：

- 现在系统已经具备“采、存、看、控”的底座
- 新增的策略能力开始真正体现 EMS 价值
- 继续补策略和测试场景，最能快速把系统从“监控后台”推进到“自动控制系统”
